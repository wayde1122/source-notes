# 第六阶段：补全数据库、企业能力与运维面

## 阶段目标

在主链路已经顺通的基础上，补完三个"侧面但很重要"的模块：
1. 数据是如何落库的（TypeORM 实体 + Migration）
2. 企业版能力是如何叠加在开源主链路之上的
3. 监控、队列、部署配置如何支撑生产运行

### 1.2 核心实体详解

**ChatFlow 实体关键字段**：

```typescript
@Entity()
class ChatFlow {
    id: string         // UUID，主键
    name: string       // 流程名称
    flowData: string   // JSON 序列化的节点图（{ nodes, edges, viewport }）
    type: ChatflowType // CHATFLOW | AGENTFLOW | MULTIAGENT | ASSISTANT
    deployed: boolean  // 是否部署（公开访问）
    isPublic: boolean  // 是否允许公开访问
    apikeyid: string   // 绑定的 API Key ID（限制访问）
    chatbotConfig: string  // 聊天机器人 UI 配置（JSON）
    analytic: string   // 可观测性配置（LangSmith/Langfuse 等）
    workspaceId: string    // 所属工作区（企业版隔离）
    createdDate / updatedDate
}
```

**ChatMessage 实体关键字段**：

```typescript
@Entity()
class ChatMessage {
    id: string
    role: 'userMessage' | 'apiMessage'
    chatflowid: string     // 关联的 ChatFlow
    content: string        // 消息正文
    sourceDocuments: string // JSON，RAG 命中的源文档
    usedTools: string      // JSON，工具调用记录
    agentReasoning: string // JSON，Agent 推理过程
    fileUploads: string    // JSON，上传的文件
    chatId: string         // 对话 ID（区分不同会话）
    sessionId: string      // Memory session ID
    executionId: string    // 关联的执行记录（AgentFlow）
}
```

### 1.3 Migration 策略

- 支持 4 种数据库各自的 Migration 目录：`sqlite/`、`mysql/`、`mariadb/`、`postgres/`
- 每次表结构变更都通过新增 Migration 文件实现，不依赖 `synchronize: true`
- 启动时执行 `AppDataSource.runMigrations({ transaction: 'each' })`，每个 Migration 独立事务
- 命令：`pnpm migration:create` 创建新的 Migration 文件

### 2.2 企业能力叠加模式

企业版能力通过以下机制叠加在开源主链路之上，而不是替换它：

**机制一：IdentityManager（身份管理器）**

```typescript
class IdentityManager {
    getPlatformType(): Platform  // OPEN_SOURCE | CLOUD | ENTERPRISE
    isLicenseValid(): boolean    // 许可证有效性
    checkFeatureByPlan(feature: string) // Feature Flag 中间件工厂
}
```

- `Platform.OPEN_SOURCE`：跳过所有认证检查，不需要登录
- `Platform.CLOUD` / `Platform.ENTERPRISE`：开启完整的 JWT 认证、RBAC、工作区隔离

**机制二：路由层 Feature Flag**

```typescript
// 企业版路由通过中间件守门
router.use('/datasets', IdentityManager.checkFeatureByPlan('feat:datasets'), datasetRouter)
router.use('/evaluations', IdentityManager.checkFeatureByPlan('feat:evaluations'), evaluationsRouter)
```

**机制三：workspaceId 数据隔离**

开源实体（ChatFlow/Credential/Tool 等）都有 `workspaceId` 字段：
- 开源模式：workspaceId 为固定默认值，不做隔离
- 企业模式：每个请求携带用户的 `activeWorkspaceId`，service 层过滤只返回当前工作区数据

**机制四：JWT Cookie 认证**

`initializeJwtCookieMiddleware()` 在企业/云模式下安装 Passport.js JWT 策略，将解码后的用户信息注入 `req.user`（含 permissions、features、工作区信息）。

### 2.3 LoggedInUser 结构

## 四、队列层（queue/）

### 4.1 队列架构

```
queue/
├── BaseQueue.ts          ← 抽象基类（BullMQ Queue 封装）
├── PredictionQueue.ts    ← 预测任务队列
├── UpsertQueue.ts        ← 向量入库任务队列
├── QueueManager.ts       ← 队列管理器（单例）
├── RedisEventPublisher.ts← Server 端发布执行事件到 Redis
└── RedisEventSubscriber.ts ← Worker 端订阅 Redis 事件
```

### 4.2 QUEUE 模式工作原理

```
[SERVER 进程]
POST /predictions/:id
    │
    ▼
PredictionQueue.addJob(jobData)
    │（写入 BullMQ / Redis）
    ▼
RedisEventPublisher.subscribe()  ← 监听执行结果
    │（通过 SSE 推送给客户端）
    ▼
客户端收到流式输出

[WORKER 进程]（独立进程）
PredictionQueue.process(job)
    │
    ▼
执行节点运行时（同直接模式）
    │
    ▼
RedisEventPublisher.publish(result) ← 发布结果到 Redis
```

### 4.3 队列管理

| 队列 | 用途 |
|------|------|
| `PredictionQueue` | 聊天/预测请求的异步执行 |
| `UpsertQueue` | 文档向量化入库任务 |

BullMQ 控制台挂载在 `/admin/queues`，可实时查看队列状态、重试失败任务。

### 4.4 Redis 配置

```
环境变量：
REDIS_URL         → 完整连接 URL（优先）
REDIS_HOST        → 主机（默认 localhost）
REDIS_PORT        → 端口（默认 6379）
REDIS_PASSWORD    → 密码
REDIS_TLS         → 是否启用 TLS
REDIS_CERT/KEY/CA → TLS 证书（Base64）
QUEUE_NAME        → 队列名称（默认 flowise-queue）
```

## 七、本阶段回答的三个问题

**Q1：数据是如何落库的？**

通过 TypeORM 实体 + `@Entity()/@Column()` 装饰器声明表结构，所有表变更通过 Migration 脚本管理，启动时自动运行待执行的 Migration。核心数据（ChatFlow 定义、消息记录、凭证）存储在关系型数据库，文件存储通过 StorageProvider 抽象层支持本地/S3/Azure/GCS。

**Q2：企业版能力是如何叠加在开源主链路之上的？**

通过三个关键机制：① `IdentityManager` 判断平台类型并决定是否开启认证；② 路由层 Feature Flag 中间件对企业功能守门；③ `workspaceId` 字段在数据层做租户隔离。企业版代码在 `enterprise/` 目录下独立维护，不侵入开源核心逻辑。

**Q3：监控、队列、Worker、部署配置如何支撑生产运行？**

- **监控**：支持 Prometheus 指标端点和 OpenTelemetry 追踪，可接入 Grafana/Jaeger 等标准可观测性平台。
- **队列**：QUEUE 模式通过 BullMQ + Redis 实现 Server 与 Worker 解耦，支持水平扩展 Worker 节点。
- **部署**：提供 docker-compose 配置覆盖单机模式和队列模式，`.env.example` 列出全部可配置项。
