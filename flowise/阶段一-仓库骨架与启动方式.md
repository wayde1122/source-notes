## 二、Monorepo 结构

```
Flowise/
├── packages/           ← 所有业务包（由 pnpm-workspace.yaml 管理）
│   ├── server/         ← 后端服务入口（Express + TypeORM）
│   ├── components/     ← 节点生态与运行时抽象（核心）
│   ├── ui/             ← 管理后台与可视化工作台（React）
│   ├── agentflow/      ← AgentFlow 画布编辑器（前端基础设施）
│   └── api-documentation/ ← 接口文档包（非核心，可后读）
├── docker/             ← 容器化部署配置
├── metrics/            ← 监控相关配置
├── assets/             ← 静态资源
├── package.json        ← 根配置（工具链、Lint、脚本入口）
└── pnpm-workspace.yaml ← 声明 packages/* 为工作区成员
```

`pnpm-workspace.yaml` 内容极简：
```yaml
packages:
    - 'packages/*'
```

## 四、启动流程解析

### 4.1 根级脚本入口

```json
"start":         "run-script-os"
"start:windows": "cd packages/server/bin && run start"
"start:default": "cd packages/server/bin && ./run start"
"dev":           "turbo run dev --parallel --no-cache"
```

- 生产启动：进入 `packages/server/bin/` 执行 `run start`。
- 开发模式：Turborepo 并行跑所有包的 `dev` 脚本（前后端同时热更新）。

### 4.2 后端启动链路

```
bin/run start
  └─> commands/start.ts (Start 类)
        └─> DataSource.init()     // 初始化数据库连接
        └─> Server.start()        // 启动 Express App
```

`commands/start.ts` 是真正的启动入口类，继承自 `BaseCommand`，按顺序调用：
1. `DataSource.init()` — 根据环境变量选择数据库驱动（SQLite / MySQL / MariaDB / PostgreSQL）
2. `Server.start()` — 初始化 Express 应用、挂载中间件、装配路由、监听端口

### 4.3 Worker 模式

```json
"start-worker":         "run-script-os"
"start-worker:windows": "cd packages/server/bin && run worker"
```

支持独立 Worker 进程，负责异步队列任务（如预测请求的队列消费）。

## 五、工具链分析

### 5.1 Turborepo

Turborepo（`turbo`）作为 monorepo 构建编排器，统一管理各包的 build / dev / test 任务的依赖关系与并行执行策略。

> 注：仓库根目录未见 `turbo.json`，配置可能内嵌于各包 `package.json` 或使用默认配置。

常用命令：
```bash
pnpm build          # turbo run build（增量构建）
pnpm build-force    # 清理后全量重建
pnpm dev            # turbo run dev --parallel（并行开发）
pnpm clean          # 清理所有包的构建产物
```

### 5.2 代码质量工具链

| 工具 | 用途 |
|------|------|
| TypeScript ^5.4.5 | 全栈类型系统 |
| ESLint ^8.24.0 | Lint（包括 React、TS、JSX）|
| Prettier ^2.7.1 | 格式化（printWidth: 140，单引号，无分号）|
| husky + lint-staged | Git commit 前自动 lint 修复 |
| pretty-quick | 只对 staged 文件格式化 |

### 5.3 依赖版本锁定（关键 AI 依赖）

通过根级 `resolutions`（Yarn 风格）+ pnpm `overrides` 锁定关键 AI/基础依赖：

| 依赖 | 锁定版本 |
|------|---------|
| `@langchain/core` | 1.1.20 |
| `openai` | 6.19.0 |
| `@anthropic-ai/sdk` | ^0.73.0 |
| `@google/generative-ai` | ^0.24.0 |
| `@grpc/grpc-js` | ^1.10.10 |
| `protobufjs` | 7.4.0 |
| `uuid` | ^10.0.0 |
| `axios` | 1.12.0（安全补丁）|

```json

```

## 六、总链路概念图

```
用户浏览器 (UI)
    │  HTTP(S)（加载前端构建产物 / 静态资源）
    ▼
packages/ui          → React 管理后台（路由、画布、表单）
    │  HTTP(S)；流式场景为 SSE（非 WebSocket）
    ▼
packages/server      → Express 路由 → Controller → Service
    │  调用节点运行时
    ▼
packages/components  → 节点抽象层 → 具体节点实现
    │  调用外部服务
    ▼
数据库 / LLM API / 向量数据库 / 工具服务
    │  结果返回
    ▼
packages/server      → 序列化响应
    │
    ▼
packages/ui          → 展示到浏览器
```

> **注**：与后端的实时流式（如内部预测）走 **`text/event-stream`（SSE）**，由 `packages/server` 侧 `SSEStreamer` 等写入、前端可用 `@microsoft/fetch-event-source` 消费。仓库依赖里虽有 `ws`（多为传递依赖/版本覆盖），**并非**浏览器与 Flowise 主 API 之间的业务 WebSocket。本地 `pnpm dev` 时，**Vite 可能另建 WebSocket 仅用于 HMR 热更新**，与上述业务协议无关。

## 七、本阶段回答的三个问题

**Q1：这是怎样的多包结构？**

典型的 pnpm + Turborepo monorepo，5 个包共用一个 `node_modules`。包之间通过 workspace 协议互相引用，turbo 管理构建顺序和缓存。

**Q2：哪些包是运行时主角，哪些是支撑层？**

- 运行时主角：`server`（后端）、`components`（节点运行时）、`ui`（前端）、`agentflow`（新画布）
- 支撑层：`api-documentation`（文档）、`docker`/`metrics`/`assets`（部署与运维）

**Q3：前后端与节点系统是如何拆开的？**

- `ui` 和 `agentflow` 是纯前端包，负责交互与可视化。
- `server` 是后端服务包，提供 **HTTP(S)** 接口；需要持续推送时主要用 **SSE**，而非对浏览器暴露 WebSocket 业务通道。
- `components` 是节点运行时包，被 `server` 调用，不直接暴露 HTTP 接口。
- 前后端通过 **REST** 与 **SSE** 等配合通信，节点系统通过 Node.js 模块边界隔离。
