## 架构总览

AnythingLLM 的文档解析采用**三层架构**，前端、Server、Collector 各司其职：

```
┌─────────────┐     HTTP      ┌──────────────┐    HTTP(:8888)    ┌──────────────────┐
│   Frontend   │ ────────────→ │    Server     │ ────────────────→ │    Collector      │
│  (React)     │               │  (Express)    │                   │  (Express:8888)   │
│              │               │               │                   │                   │
│ · 拖拽上传    │               │ · Multer 接收  │                   │ · 文本抽取         │
│ · 链接输入    │               │ · CollectorApi │                   │ · 格式转换         │
│ · 文档管理    │               │ · 分块/Embed   │                   │ · Token 估算       │
│ · 嵌入触发    │               │ · 向量存储      │                   │ · 写入 JSON        │
└─────────────┘               └──────────────┘                   └──────────────────┘
                                     │
                              ┌──────┴──────┐
                              │  向量数据库   │
                              │ (LanceDB等)  │
                              └─────────────┘
```

**核心设计原则**：前端不直接与 Collector 通信，所有请求经 Server 中转，Collector 作为纯粹的文档处理微服务。

## 一、前端上传入口

前端提供三种文档输入方式，最终都通过 Server API 进入处理流水线。

### 1.1 文档管理弹窗上传

**组件**：`frontend/src/components/Modals/ManageWorkspace/Documents/UploadFile/index.jsx`

用户在 Workspace 设置中打开文档管理弹窗，通过 `react-dropzone` 拖拽或点击上传文件：

```
用户拖拽文件 → FileUploadProgress 组件
            → Workspace.uploadFile(slug, formData)
            → POST /api/workspace/:slug/upload
```

上传前会调用 `System.checkDocumentProcessorOnline()` 检查 Collector 服务是否在线。

**链接抓取**也在此处：

```
用户输入 URL → Workspace.uploadLink(slug, link)
             → POST /api/workspace/:slug/upload-link
```

### 1.2 聊天区域拖拽上传（DnD）

**组件**：`frontend/src/components/WorkspaceChat/ChatContainer/DnDWrapper/index.jsx`

聊天界面支持拖拽/粘贴文件作为对话附件：

```
拖拽文件到聊天区 → 区分类型：
  · 图片 → Base64 编码，作为 attachment 直接嵌入
  · 文档 → Workspace.parseFile(slug, formData)
         → POST /api/workspace/:slug/parse
         → 解析后返回 token 估算
         → 用户确认后嵌入或放弃
```

这里有一个**智能决策流程**：当解析后的文档 token 数超过 context window 时，弹出模态框让用户选择：
- **继续**：直接作为上下文使用
- **嵌入**：走完整的 Embedding 流程存入向量库
- **放弃**：取消操作

### 1.3 嵌入触发

文档上传后存放在 Server 的 `documents/` 目录，用户需要手动将文档"移入"工作区才会触发 Embedding：

```
用户在文档管理中将文档拖入工作区
  → Workspace.modifyEmbeddings(slug, { adds: [...], deletes: [...] })
  → POST /api/workspace/:slug/update-embeddings
  → Document.addDocuments() / Document.removeDocuments()
```

### 1.4 前端 API 模型汇总

| 模型文件 | 方法 | API 路径 |
|---------|------|---------|
| `workspace.js` | `uploadFile()` | `POST /api/workspace/:slug/upload` |
| `workspace.js` | `parseFile()` | `POST /api/workspace/:slug/parse` |
| `workspace.js` | `uploadLink()` | `POST /api/workspace/:slug/upload-link` |
| `workspace.js` | `modifyEmbeddings()` | `POST /api/workspace/:slug/update-embeddings` |
| `workspace.js` | `uploadAndEmbedFile()` | `POST /api/workspace/:slug/upload-and-embed` |
| `workspace.js` | `embedParsedFile()` | `POST /api/workspace/:slug/embed-parsed-file/:id` |
| `workspace.js` | `deleteParsedFiles()` | `DELETE /api/workspace/:slug/delete-parsed-files` |

## 二、Server 层中转

### 2.1 文件接收（Multer）

**文件**：`server/utils/files/multer.js`

Server 使用 Multer 中间件接收上传文件，存储到 Collector 的 `hotdir` 目录：

```
POST /api/workspace/:slug/upload
  → handleFileUpload (Multer)
  → 文件落盘到 collector/hotdir/
  → CollectorApi.processDocument(originalname, metadata)
```

### 2.2 CollectorApi 桥接

**文件**：`server/utils/collectorApi/index.js`

`CollectorApi` 是 Server 与 Collector 之间的通信桥梁，封装了所有对 Collector 的 HTTP 调用：

```javascript
class CollectorApi {
  constructor() {
    this.comkey = new CommunicationKey();
    this.endpoint = `http://0.0.0.0:${process.env.COLLECTOR_PORT || 8888}`;
  }

  // 处理文档文件
  async processDocument(filename, metadata) { ... }

  // 处理链接
  async processLink(link, scraperHeaders, metadata) { ... }

  // 处理原始文本
  async processRawText(textContent, metadata) { ... }

  // 仅解析不入库
  async parseDocument(filename) { ... }

  // 转发扩展请求（GitHub/YouTube/Confluence 等）
  async forwardExtensionRequest({ endpoint, method, body }) { ... }
}
```

每个请求都会附带：
- **`X-Integrity`**：RSA-SHA256 签名，验证请求完整性
- **`X-Payload-Signer`**：加密的 payload signer，用于扩展中的敏感数据解密
- **`options`**：运行时配置（Whisper 提供商、OCR 语言等）

### 2.3 两种处理模式

| 模式 | API | 说明 |
|------|-----|------|
| **process** | `POST /process` | 解析文档并写入 `documents/custom-documents/` |
| **parse** | `POST /parse` | 仅解析文档，写入 `direct-uploads/`，不进入文档选择器 |

`parse` 模式用于聊天附件场景——文件被解析后临时使用，不需要持久化到文档库。

## 三、Collector 文档处理服务

### 3.1 服务架构

**入口**：`collector/index.js`

Collector 是一个独立的 Express 服务，监听端口 8888：

```
collector/
├── index.js                    # Express 主服务
├── hotdir/                     # 文件上传临时目录
├── storage/tmp/                # 临时处理目录
├── processSingleFile/          # 单文件处理核心
│   ├── index.js                # 路由分发
│   └── convert/                # 各格式转换器
├── processLink/                # 链接处理
├── processRawText/             # 原始文本处理
├── extensions/                 # 扩展数据源路由
├── middleware/                 # 中间件
└── utils/                      # 工具库
```

**启动时行为**：调用 `wipeCollectorStorage()` 清空 `hotdir` 和 `tmp` 目录，防止残留大文件占用磁盘。

### 3.2 路由表

| 路由 | 方法 | 处理函数 | 说明 |
|------|------|---------|------|
| `/process` | POST | `processSingleFile()` | 处理单文件并写入文档库 |
| `/parse` | POST | `processSingleFile(, {parseOnly: true})` | 仅解析，写入 direct-uploads |
| `/process-link` | POST | `processLink()` | 处理 URL 链接 |
| `/process-raw-text` | POST | `processRawText()` | 处理原始文本 |
| `/util/get-link` | POST | `getLinkText()` | 获取链接文本（不保存） |
| `/accepts` | GET | - | 返回支持的 MIME 类型列表 |
| `/ext/*` | POST | 各扩展处理器 | 扩展数据源 |

### 3.3 单文件处理核心流程

**文件**：`collector/processSingleFile/index.js`

```
processSingleFile(targetFilename, options, metadata)
  │
  ├─ 1. 路径解析：WATCH_DIRECTORY + normalizePath(targetFilename)
  │
  ├─ 2. 安全校验
  │     ├─ isWithin() 防目录穿越
  │     ├─ RESERVED_FILES 检查（__HOTDIR__.md 不可处理）
  │     └─ 文件存在性检查
  │
  ├─ 3. 扩展名识别
  │     ├─ 有扩展名 → 查 SUPPORTED_FILETYPE_CONVERTERS
  │     ├─ 不支持但是文本类型 → isTextType() → 当作 .txt 处理
  │     └─ 不支持且非文本 → trashFile() → 返回失败
  │
  ├─ 4. 动态加载转换器
  │     └─ require(SUPPORTED_FILETYPE_CONVERTERS[processFileAs])
  │
  └─ 5. 执行转换
        └─ FileTypeProcessor({ fullFilePath, filename, options, metadata })
        └─ 返回 { success, reason, documents }
```

### 3.4 文件类型判定

**MIME 检测**：`collector/utils/files/mime.js` 中的 `MimeDetector`

**文本类型判定**：`collector/utils/files/index.js` 中的 `isTextType()`

```
isTextType(filepath)
  │
  ├─ isKnownTextMime(filepath)
  │   ├─ 获取 MIME 类型
  │   ├─ 检查是否在 badMimes 列表中 → false
  │   ├─ 检查主类型是否在 nonTextTypes 中 → false
  │   └─ 通过 → true
  │
  └─ parseableAsText(filepath)  // MIME 未知时的降级方案
      ├─ 读取文件前 1KB
      ├─ 统计 null 字符和控制字符数量
      └─ 占比 < 10% → 判定为文本文件
```

这个设计确保了即使文件扩展名未知（如 `.ts`、`.tsx`、`.vue` 等），只要内容是文本就能正确处理。

## 四、各格式转换器详解

所有转换器遵循统一接口，输入输出格式一致：

```javascript
// 输入
async function converter({ fullFilePath, filename, options, metadata }) { ... }

// 输出
{
  success: boolean,
  reason: string | null,
  documents: [{
    id: string,              // UUID v4
    url: string,             // file:// 协议路径
    title: string,           // 文档标题
    docAuthor: string,       // 作者
    description: string,     // 描述
    docSource: string,       // 来源说明
    chunkSource: string,     // 块来源标识
    published: string,       // 创建日期
    wordCount: number,       // 词数
    pageContent: string,     // 文本内容（核心）
    token_count_estimate: number,  // Token 估算
    location: string,        // 相对路径（writeToServerDocuments 添加）
  }]
}
```

### 4.1 asTxt.js — 文本文件处理

**路径**：`collector/processSingleFile/convert/asTxt.js`

**处理的格式**：`.txt` `.md` `.org` `.adoc` `.rst` `.csv` `.json` `.html`

**流程**：

```
读取文件 (fs.readFileSync, utf8)
  → 检查内容是否为空
  → 构建文档对象 { id, url, title, pageContent, token_count_estimate, ... }
  → writeToServerDocuments() 写入 JSON
  → trashFile() 删除原始文件
  → 返回 { success: true, documents: [document] }
```

这是最简单的转换器——直接读取文件内容作为 `pageContent`，不做任何额外处理。

### 4.2 asPDF — PDF 处理

**路径**：`collector/processSingleFile/convert/asPDF/index.js`

**依赖**：`pdf-parse`（PDFLoader）、`Tesseract.js`（OCRLoader）

**流程**：

```
PDFLoader.load() 提取文本
  │
  ├─ 成功 → 按页合并 pageContent
  │
  └─ 失败（无文本，可能是扫描件）
       → OCRLoader.ocrPDF() 使用 Tesseract OCR
       → 提取图片中的文字
```

**关键设计**：PDF 处理有**降级策略**——先尝试文本提取，失败后自动切换到 OCR。这确保了扫描版 PDF 也能被正确处理。

```javascript
let docs = await pdfLoader.load();
if (docs.length === 0) {
  docs = await new OCRLoader({
    targetLanguages: options?.ocr?.langList,  // 可配置 OCR 语言
  }).ocrPDF(fullFilePath);
}
```

### 4.3 asDocx.js — Word 文档处理

**依赖**：LangChain 的 `DocxLoader`

直接使用 LangChain 社区加载器提取 `.docx` 文本内容。

### 4.4 asOfficeMime.js — Office 通用处理

**处理的格式**：`.pptx` `.odt` `.odp`

**依赖**：`officeparser`

使用 officeparser 库统一处理 Microsoft Office 和 OpenDocument 格式。

### 4.5 asXlsx.js — Excel 处理

**依赖**：`node-xlsx`

**特殊处理**：每个 Sheet 生成一个独立文档，这是唯一一个可能返回多个 document 的转换器。

### 4.6 asAudio.js — 音频/视频处理

**处理的格式**：`.mp3` `.wav` `.mp4` `.mpeg`

**依赖**：
- `localWhisper`：本地 `@xenova/transformers` 模型
- `OpenAiWhisper`：OpenAI Whisper API
- `ffmpeg`：音频格式转换（转为 WAV）

**流程**：

```
音频文件 → ffmpeg 转码为 WAV
         → Whisper 语音识别
         → 提取文本作为 pageContent
```

Whisper 提供商由 `options.whisperProvider` 控制，默认使用本地模型。

### 4.7 asImage.js — 图片 OCR 处理

**处理的格式**：`.png` `.jpg` `.jpeg` `.webp`

**依赖**：Tesseract.js

```
图片文件 → OCRLoader.ocrImage()
         → Tesseract OCR 识别
         → 提取文字作为 pageContent
```

支持多语言 OCR，通过 `options.ocr.langList` 配置（默认 `eng`）。

### 4.8 asMbox.js — 邮箱归档处理

**依赖**：`mbox-parser`

解析 `.mbox` 邮箱归档文件，提取邮件内容。

### 4.9 asEPub.js — 电子书处理

**依赖**：LangChain 的 `EPubLoader`

解析 `.epub` 电子书格式。

## 五、文档写入与存储

### 5.1 writeToServerDocuments

**文件**：`collector/utils/files/index.js`

所有转换器最终都调用此函数将解析结果写入磁盘：

```javascript
function writeToServerDocuments({ data, filename, destinationOverride, options }) {
  // 确定写入目标
  let destination;
  if (destinationOverride)
    destination = path.resolve(destinationOverride);
  else if (options.parseOnly)
    destination = path.resolve(directUploadsFolder);  // direct-uploads/
  else
    destination = path.resolve(documentsFolder, "custom-documents");  // documents/custom-documents/

  // 写入 JSON 文件
  fs.writeFileSync(destinationFilePath, JSON.stringify(data, null, 4));

  return {
    ...data,
    location: destinationFilePath.split("/").slice(-2).join("/"),  // 相对路径
    isDirectUpload: options.parseOnly || false,
  };
}
```

### 5.2 存储路径

```
server/storage/
├── documents/
│   └── custom-documents/        # 正常上传的文档 JSON
│       ├── filename-uuid.json
│       └── ...
├── direct-uploads/              # parseOnly 模式的临时文档
│   ├── filename-uuid.json
│   └── ...
└── vector-cache/                # 向量缓存
    ├── filename-uuid.json
    └── ...
```

### 5.3 文档 JSON 结构

每个文档被序列化为一个 JSON 文件：

```json
{
    "id": "123e4567-e89b-12d3-a456-426614174000",
    "url": "file://D:\\anything-llm\\collector\\hotdir\\example.pdf",
    "title": "example.pdf",
    "docAuthor": "John Doe",
    "description": "A sample document",
    "docSource": "pdf file uploaded by the user.",
    "chunkSource": "",
    "published": "3/17/2026, 10:30:00 AM",
    "wordCount": 1500,
    "pageContent": "The full text content of the document...",
    "token_count_estimate": 2048
}
```

### 5.4 Token 估算

**文件**：`collector/utils/tokenizer/index.js`

使用 `js-tiktoken`（`cl100k_base` 编码）进行 Token 计数：

```javascript
class TikTokenTokenizer {
  static MAX_KB_ESTIMATE = 10;  // 超过 10KB 使用估算
  static DIVISOR = 8;           // 估算时的字符/token 比率

  tokenizeString(input) {
    if (this.#isTooLong(input)) {
      // 大文本使用粗略估算，避免 CPU 阻塞
      return Math.ceil(input.length / TikTokenTokenizer.DIVISOR);
    }
    return this.encoder.encode(input).length;
  }
}
```

**设计亮点**：对超过 10KB 的文本使用 `length / 8` 的粗略估算，避免 tiktoken 编码大文本时的 CPU 开销。采用单例模式确保编码器只初始化一次。

## 六、分块（Chunking）

### 6.1 TextSplitter

**文件**：`server/utils/TextSplitter/index.js`

分块发生在 Server 端（不是 Collector），当文档被添加到 Workspace 时触发。

```javascript
class TextSplitter {
  constructor(config = {}) {
    // config: { chunkPrefix, chunkSize, chunkOverlap, chunkHeaderMeta }
    this.#splitter = new RecursiveSplitter({
      chunkSize: config?.chunkSize ?? 1000,
      chunkOverlap: config?.chunkOverlap ?? 20,
      chunkHeader: this.stringifyHeader(),
    });
  }

  static determineMaxChunkSize(preferred, embedderLimit = 1000) {
    // 取用户偏好和 Embedder 限制的较小值
    const prefValue = isNullOrNaN(preferred) ? Number(embedderLimit) : Number(preferred);
    const limit = Number(embedderLimit);
    return prefValue > limit ? limit : prefValue;
  }
}
```

### 6.2 分块策略

底层使用 LangChain 的 `RecursiveCharacterTextSplitter`，按以下分隔符递归切分：

```
"\n\n" → "\n" → " " → ""
```

优先在段落边界切分，其次在换行处，最后在空格处，确保语义完整性。

### 6.3 Chunk Header 元数据

每个 chunk 可以附加元数据头，帮助 LLM 理解上下文来源：

```javascript
static buildHeaderMeta(metadata) {
  // 从文档元数据中提取：
  // - sourceDocument: 文档标题
  // - published: 发布日期
  // - source: 原始 URL（仅 link:// 和 youtube:// 前缀）
}

stringifyHeader() {
  // 生成格式：
  // <document_metadata>
  // sourceDocument: example.pdf
  // published: 3/17/2026
  // source: https://example.com
  // </document_metadata>
}
```

### 6.4 Chunk Size 决策

```
用户配置的 chunkSize（系统设置）
        ↓
TextSplitter.determineMaxChunkSize(preferred, embedderLimit)
        ↓
取 min(preferred, embedderLimit)
        ↓
实际 chunkSize
```

Embedder 模型有硬性 token 限制，chunkSize 不能超过该限制。

## 七、Embedding 向量化

### 7.1 触发时机

Embedding 在 `Document.addDocuments()` 中触发，即用户将文档添加到 Workspace 时：

```javascript
// server/models/documents.js
addDocuments: async function (workspace, additions) {
  const VectorDb = getVectorDbClass();

  for (const path of additions) {
    const data = await fileData(path);  // 读取文档 JSON

    const { vectorized, error } = await VectorDb.addDocumentToNamespace(
      workspace.slug,    // namespace = workspace slug
      { ...data, docId },
      path
    );

    if (vectorized) {
      await prisma.workspace_documents.create({ data: newDoc });
    }
  }
}
```

### 7.2 向量化流程（以 LanceDB 为例）

```
VectorDb.addDocumentToNamespace(namespace, documentData, fullFilePath)
  │
  ├─ 1. 检查缓存：cachedVectorInformation(fullFilePath)
  │     ├─ 有缓存 → 直接使用缓存的向量数据写入向量库
  │     └─ 无缓存 → 继续处理
  │
  ├─ 2. 分块：TextSplitter.splitText(pageContent)
  │     → 返回 string[] 文本块数组
  │
  ├─ 3. Embedding：EmbedderEngine.embedChunks(textChunks)
  │     → 返回 float[][] 向量数组
  │
  ├─ 4. 写入向量库
  │     → 按批次写入（每批约 500 条）
  │
  └─ 5. 缓存：storeVectorResult(vectors, fullFilePath)
        → 写入 vector-cache/ 目录
```

### 7.3 向量缓存机制

**目的**：同一文档被添加到多个 Workspace 时，避免重复 Embedding。

```
第一次嵌入：文档 → 分块 → Embedding → 写入向量库 + 写入 vector-cache
第二次嵌入：文档 → 读取 vector-cache → 直接写入向量库（跳过分块和 Embedding）
```

缓存文件存储在 `server/storage/vector-cache/` 目录。

```
scrapeGenericUrl({ link, captureAs, scraperHeaders, metadata, saveAsDocument })

  │     └─ processVia: 'web' | 'file' | 'youtube'
```

## 十、安全机制

### 10.1 请求完整性校验

**文件**：`collector/middleware/verifyIntegrity.js`

Collector 不对外暴露，但仍通过签名机制确保请求来自合法的 Server：

```
Server 端：
  data = JSON.stringify(requestBody)
  signature = comkey.sign(data)  // RSA-SHA256 签名
  headers['X-Integrity'] = signature

Collector 端：
  verifyPayloadIntegrity(req, res, next)
    → CommunicationKey.verify(data, signature)
    → 验证通过 → next()
    → 验证失败 → 403
```

开发环境（`NODE_ENV === "development"`）跳过签名校验。

### 10.2 数据加密

**文件**：`collector/middleware/setDataSigner.js`

对于需要处理敏感数据的扩展（如 Confluence API Token），使用 AES-256-CBC 加密：

```
Server 端：
  xPayload = EncryptionManager.xPayload  // 加密密钥
  encrypted = comkey.encrypt(xPayload)
  headers['X-Payload-Signer'] = encrypted

Collector 端：
  setDataSigner(req, res, next)
    → 解密 X-Payload-Signer
    → 设置 response.locals.encryptionWorker
    → 扩展可用 encryptionWorker 解密 chunkSource 中的敏感参数
```

### 10.3 路径安全

```javascript
// 防止目录穿越
function isWithin(outer, inner) {
  if (outer === inner) return false;
  const rel = path.relative(outer, inner);
  return !rel.startsWith("../") && rel !== "..";
}

// 路径规范化，移除 ../ 前缀
function normalizePath(filepath) {
  return path.normalize(filepath.trim())
    .replace(/^(\.\.(\/|\\|$))+/, "")
    .trim();
}
```
