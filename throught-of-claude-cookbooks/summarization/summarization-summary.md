# Summarization 技术要点总结

来源：`D:\claude-cookbooks\capabilities\summarization\guide_zh.ipynb`

## 1. 核心目标

这个 cookbook 演示如何用 Claude 对长文本，尤其是法律类文档，进行高质量摘要。它关注的不是“让模型简单总结一下”，而是如何把摘要任务做成一个可控、可解析、可评估、可迭代优化的工程流程。

输入主要是 PDF 或文本格式的转租协议；输出则是结构化摘要，例如参与方、物业信息、租期和租金、责任划分、通知与同意条款、特殊约定等。这个方案适合合同审阅、尽调、文档管理、企业知识库摘要、长文档信息抽取等场景。

摘要任务的难点在于：好的摘要并不只有一个标准答案。法律文档还要求事实准确、条款完整、格式稳定，并能明确指出未在原文中说明的信息。因此 notebook 逐步从基础 prompt 发展到 few-shot、领域引导、长文档分块、摘要索引 RAG 和 Promptfoo 评估。

---

## 2. 技术架构

整体流程可以概括为：

```text
PDF / 文本文档
  ↓
文本抽取与清洗
  ↓
Prompt 策略选择
  ├─ 基础摘要
  ├─ Few-shot 摘要
  ├─ 引导式法律摘要
  ├─ 领域特定转租协议摘要
  └─ 长文档分块 + 元摘要
  ↓
Claude 生成结构化摘要
  ↓
XML 标签解析 / 后处理
  ↓
Promptfoo + 自定义指标评估
  ↓
基于失败样例迭代 prompt
```

各组件职责：

| 组件 | 职责 |
|---|---|
| PDF 解析 | 用 `pypdf` 从法律 PDF 中抽取文本 |
| 文本清洗 | 去除多余空白、页码，并按 token 限制截断 |
| Claude 摘要函数 | 通过不同 prompt 策略生成摘要 |
| XML 输出约束 | 让摘要变成可解析结构，而不是自由文本 |
| 长文档分块 | 对超长文档先分段摘要，再汇总成元摘要 |
| Summary Indexed Documents | 先为文档建摘要索引，再按查询排序和抽取条款 |
| Promptfoo 评估 | 用格式检查、ROUGE、BLEU、LLM-as-judge 等方式评估 |

这个设计的核心思想是：**摘要不只是生成任务，也是信息抽取、结构化输出和评估体系的组合工程**。

## 3. 核心流程

Notebook 展示了摘要方案的逐级演进：

| 阶段 | 方法 | 解决的问题 |
|---|---|---|
| 基础摘要 | 直接要求 Claude 用 bullet points 总结 | 快速得到可读摘要 |
| Few-shot 摘要 | 在 prompt 中加入多个转租协议摘要示例 | 提高格式一致性和领域适配性 |
| 引导式摘要 | 明确要求关注参与方、主题、条款、日期、特殊条款 | 减少遗漏，提高可审查性 |
| 领域特定摘要 | 针对转租协议定义固定字段和 XML 结构 | 生成可解析、可比较的结构化摘要 |
| 长文档摘要 | 分块摘要后再合并成元摘要 | 处理超出上下文或多文档汇总场景 |
| 摘要索引 RAG | 为文档生成摘要，按查询相关性排序，再抽取原文条款 | 支持多文档检索和精准条款定位 |

基础摘要虽然容易实现，但输出格式不稳定，也难以判断是否遗漏关键条款。few-shot 通过示例让模型模仿目标格式；引导式 prompt 则显式规定摘要维度。对法律文档而言，后者尤其重要，因为用户通常关心固定字段，而不是泛泛的“主要内容”。

## 4. 关键实现点

### 4.1 文档预处理

预处理函数包括：

- `extract_text_from_pdf`：从 PDF 每页抽取文本；
- `clean_text`：压缩空白、去除页码；
- `prepare_for_llm`：按近似 token 限制截断文本；
- `get_llm_text`：封装整个读取与清洗流程。

这一步看似简单，但对摘要质量影响很大。PDF 抽取文本常包含换行、页码、断词和格式噪声，如果不清理，会增加模型理解成本。

### 4.2 Prompt 约束与 assistant 预填

摘要函数普遍使用：

- `system` 指定角色，例如“法律分析师”或“房地产法律专家”；
- user prompt 描述摘要维度和输出要求；
- assistant 预填 `Here is the summary... <summary>`；
- `stop_sequences=["</summary>"]` 控制模型在目标标签结束处停止。

这种模式可以提升输出边界的稳定性，便于后续解析。

### 4.3 XML 结构化输出

领域特定摘要要求输出：

- `<parties involved>`
- `<property details>`
- `<term and rent>`
- `<responsibilities>`
- `<consent and notices>`
- `<special provisions>`

然后用正则 `r"<(.*?)>(.*?)</\1>"` 解析为字典。这里的工程价值在于：摘要结果可以进入下游系统，例如审阅界面、合同数据库、差异对比、自动检查器，而不是只供人阅读。

### 4.4 长文档分块与元摘要

长文档策略是：先按字符长度切块，对每个 chunk 使用较便宜的 Haiku 生成中间摘要，再把所有中间摘要交给 Sonnet 生成最终摘要。

```text
长文档
  ↓
chunk_1, chunk_2, ...
  ↓
Haiku 生成局部摘要
  ↓
Sonnet 合并为全局摘要
```

这种方法的 tradeoff 是：成本和上下文压力降低，但如果 chunk 切分粗糙，跨段关系可能丢失。因此实际生产中应按章节、标题、页码或语义边界切分，而不是单纯按字符数切分。

### 4.5 摘要索引文档

`LegalSummaryIndexedDocuments` 类展示了面向多文档检索的设计：

1. `add_document` 加入文档；
2. `generate_summaries` 为每份文档生成领域摘要；
3. `rank_documents` 用 Haiku 给摘要与查询的相关性打 0–10 分；
4. `extract_relevant_clauses` 从最高相关文档中抽取原始条款。

这个模式与普通 RAG 的差异是：先用摘要做文档级排序，再用原文做条款级抽取。摘要用于降低上下文和排序成本，原文用于保留法律语言的精确性。

## 5. 指标与结果

该目录包含 Promptfoo 评估配置，评估对象包括：

- `basic_summarize`
- `guided_legal_summary`
- `summarize_long_document`

测试数据是 9 份样例转租协议及对应参考摘要。断言包括：

| 评估方式 | 作用 |
|---|---|
| `icontains-all` | 检查是否包含固定摘要章节 |
| `llm_eval.py` | 用 LLM 评估连贯性、相关性、事实准确性等 |
| `rouge_eval.py` | 比较生成摘要与参考摘要的重叠程度 |
| `bleu_eval.py` | 衡量生成文本与参考文本的相似度 |
| `not-contains-any` | 避免输出 “I am an AI” 等无关套话 |

Notebook 中给出的观察是：Sonnet 版本整体表现更好，其中最佳表现者在评估中约有 66% 通过率，45 个测试中只有 3 个失败。作者也强调这只是示例数据上的初始结果，真实业务中应基于真实文档和审阅标准继续细化评估。

摘要评估的关键限制是：ROUGE/BLEU 更适合衡量文本重叠，却不能充分判断法律摘要是否事实准确、是否遗漏关键风险。因此更合理的做法是把格式检查、传统指标、LLM-as-judge 和人工抽检结合起来。

## 6. 技术亮点

1. **从自由文本摘要演进到结构化摘要**  
   XML 标签让结果可解析、可比较、可进入下游流程。

2. **Few-shot 与领域引导结合**  
   示例帮助模型学习格式，领域字段帮助模型聚焦法律要点。

3. **用 assistant 预填和 stop sequence 控制输出边界**  
   这比只在 prompt 里说“请用 XML 输出”更稳定。

4. **长文档采用 map-reduce 式摘要**  
   先局部摘要，再全局合并，适合多文档和超长文档。

5. **摘要索引用于文档级检索**  
   用短摘要做排序，用原文做最终抽取，在成本和准确性之间取得平衡。

6. **评估驱动 prompt 迭代**  
   通过 Promptfoo 把 prompt、模型、测试集和指标统一管理，避免只凭主观感受调 prompt。

---

## 7. 局限与注意点

- 示例中 API key 写法是 demo 形式，生产中应使用 `.env` 和环境变量管理密钥。
- 简单字符分块可能破坏法律条款边界，应改为语义或结构化分块。
- XML 解析依赖模型严格遵守格式，真实系统应增加 JSON schema、重试或校验逻辑。
- 摘要索引排序使用模型打分，成本和延迟会随文档数量增长。
- ROUGE/BLEU 对法律摘要质量的解释力有限，不能替代专家审核。
- Notebook 是教学 demo，缺少生产级异常处理、缓存、权限控制、审计日志和隐私保护。

---

## 8. 工程启发

1. **先定义摘要字段，再让模型生成**：对合同、财报、病历等高风险文档，结构化字段比泛化摘要更可靠。
2. **把摘要结果设计成数据结构**：XML/JSON 输出能让 LLM 结果进入自动化系统，而不只是显示给用户。
3. **用评估集管理 prompt 演进**：每次 prompt 修改都应在固定样例上回归测试。
4. **长文档不要只依赖大上下文**：分块、局部摘要、全局合并和检索排序仍然是重要工程手段。
5. **成本优化可以分层做**：用 Haiku 做中间摘要或排序，用 Sonnet 做最终高价值生成，是一个可迁移的模式。
